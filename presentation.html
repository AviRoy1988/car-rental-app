üîç Key Design Decisions

1. Strategy Pattern for Pricing

  • Why? Open/Closed Principle - Add new categories without modifying existing code
  • Benefit: Each calculator has single responsibility and is independently testable
  • Future-proof: Easy to add new car categories

2. GUID for Booking Numbers

  • Security: Prevents enumeration attacks
  • Distributed Systems: Can generate unique IDs without central coordination
  • Scalability: Works across multiple instances/databases

3. Repository Pattern

  • Abstraction: Decouples business logic from data access
  • Testability: Easy to mock for unit testing
  • Flexibility: Can swap data sources without changing business logic

üìù Assumptions Made

Business Logic

  • Day Calculation: Full 24-hour periods (ceiling division)
  • Distance Calculation: Simple subtraction (return - pickup)
  • Email Required: Added for invoice delivery (practical business need)
  • Single Active Rental: One car can only have one active rental at a time
  • Price Rounding: 2 decimal places (standard currency precision)

Technical Architecture

  • Controller-Based API: Better organization than minimal API
  • Database Persistence: SQL Server for production-ready storage
  • Global Exception Handling: Consistent error responses
  • Validation at API Layer: DataAnnotations for request validation

üîí Data Validation

Pickup Request Validation

  • Registration Number: Required, max 20 characters
  • SSN: Required, max 20 characters
  • Email: Required, valid email format
  • Category: Must be valid enum (0, 1, or 2)
  • Pickup Date: Required, valid datetime
  • Meter Reading: Required, must be positive

Return Request Validation

  • Booking Number: Must exist, rental must be Active
  • Return Date: Required, must be after pickup date
  • Meter Reading: Required, must be > pickup reading

Error Handling

  • Global exception middleware catches all errors
  • Returns appropriate HTTP status codes (400, 404, 500)
  • Structured error responses with clear messages
